# Documents

## Search Documents

```yaml
tool: mcp__feishu-mcp__docx_builtin_search
data:
  search_key: "产品方案"
  count: 10
useUAT: true
```

## Read Raw Content

```yaml
tool: mcp__feishu-mcp__docx_v1_document_rawContent
path:
  document_id: "doxcnxxxxxx"
useUAT: true
```

## Create And Populate A Document

```yaml
tool: mcp__feishu-mcp__docx_v1_document_create
data:
  title: "Project Report"
useUAT: true
```

Then append blocks to the returned `document_id`:

```yaml
tool: mcp__feishu-mcp__docx_v1_documentBlockChildren_create
path:
  document_id: "doxcnxxxxxx"
  block_id: "doxcnxxxxxx"
data:
  children:
    - block_type: 2
      text:
        elements:
          - text_run:
              content: "Generated by Codex."
useUAT: true
```

## Docs/Wiki To Write-Back

Use this when Codex needs to turn Feishu knowledge into a shareable result:

1. Search Docs or Wiki.
2. Read the selected content.
3. Summarize or transform the content with Codex.
4. Create a new Docx document and append the generated blocks.
5. Resolve its URL with `drive_v1_meta_batchQuery` and send the reference back to Feishu IM.

See `examples/docs-wiki-to-doc.md`.

Minimum permissions:

- `docx:document` for Docs search/read workflows.
- `wiki:wiki:readonly` for Wiki search/read workflows.
- `docx:document` for document creation and block write-back.
- `drive:drive` for metadata and the final document URL.

Read-only path:

- Search Docs
- Search Wiki
- Read Docx raw content

Write-back path:

- All read-only steps above
- `docx_v1_document_create`
- `docx_v1_documentBlockChildren_create`
- `drive_v1_meta_batchQuery`
- Optional IM push of the new document reference

Use `useUAT: true` when the current user should own and directly open the created document. Read-only exploration can stay in user token mode or the token mode already required by the workflow.

## Notes

- Prefer user token mode for documents the user should open directly.
- Document URLs often include the document ID after `/docx/`.
- Wiki pages may point to underlying document tokens. Resolve the Wiki node before reading content.

## 中文说明

### 搜索与读取文档

`docx_builtin_search` 用于按关键词搜索文档，`docx_v1_document_rawContent` 用于读取 Docx 原始内容。
如果用户最终要直接打开文档，优先使用 user token mode。

### 创建并写入文档

先用 `docx_v1_document_create` 创建新版文档，再用 `docx_v1_documentBlockChildren_create` 写入结构化块，最后通过 `drive_v1_meta_batchQuery` 获取可点击链接。

### Docs / Wiki 写回路径

推荐流程：

1. 搜索 Docs 或 Wiki
2. 读取选中的文档内容
3. 让 Codex 生成总结或结构化输出
4. 创建新版 Docx 并写入结构化块
5. 需要时再把文档引用发回飞书 IM

最小权限区分：

- 只读路径：
  - 搜索 Docs
  - 搜索 Wiki
  - 读取 Docx 内容
- 写回路径：
  - 上面的只读步骤
  - `docx_v1_document_create`
  - `docx_v1_documentBlockChildren_create`
  - `drive_v1_meta_batchQuery`
  - 可选的 IM 推送

权限要求：

- `docx:document`：Docs 搜索和读取
- `wiki:wiki:readonly`：Wiki 搜索和读取
- `docx:document`：创建文档和写入块
- `drive:drive`：查询文档元数据和链接

`useUAT: true` 的使用建议：

- 如果创建后的文档需要由当前用户直接打开或持有，使用 `useUAT: true`
- 只读探索流程可以沿用当前 workflow 已经要求的 token mode

### 额外说明

- 文档 URL 里通常会包含 `/docx/` 后面的 document ID
- Wiki 页面很多时候只是入口，真正内容还是要落到对应的 Docx token 上再读取
