# Feishu for Codex v0.3.0-beta

Beta release focused on turning the plugin from a local demo into a controllable, long-running Feishu collaboration entry point.

## Added

- Unified CLI entry: `npm run feishu -- <command>`.
- Long-connection bot bridge for `message -> runner -> downstream command -> reply`.
- Repo-provided runner for stable Feishu-to-Codex payload normalization.
- Repo-provided echo command for local bridge verification before calling real Codex.
- Session-scoped state persistence by `chat_id` or `open_id`.
- Minimal command set in chat:
  - `/help`
  - `/new`
  - `/status`
  - `/stop`
  - `/cd <path>`
- Access control:
  - owner
  - admins
  - allowed users
  - allowed chats
- Serial per-session queue with optional short batching window.
- Duplicate Feishu `message_id` delivery guard to avoid repeated local execution and repeated replies.
- macOS-first service management:
  - `start`
  - `stop`
  - `restart`
  - `status`
- `launchd` plist generation and service metadata persistence under the bot state directory.

## Bot Bridge Capabilities

- Receives Feishu long-connection text messages.
- Isolates session state by chat or sender identity.
- Runs local downstream commands through a stable runner contract.
- Supports `stdin` and `env` payload modes.
- Persists queue, workspace, run status, and last task result.
- Returns readable access denied, queued, cancelled, failed, and completed replies.
- Ignores duplicate message deliveries within the recent per-session message window.

## Service Management

- Generates `service/launchd.plist` under the bot state directory.
- Writes `service/service.json` with label, mode, log paths, and timestamps.
- Uses `service/stdout.log` and `service/stderr.log` by default.
- Treats `launchctl` as the source of truth for loaded/running state.

## Verification

Run:

```bash
scripts/smoke-test.sh
```

This validates:

- plugin JSON and required files
- executable script permissions
- MCP `initialize` and `tools/list`
- webhook self-test and fixtures
- runner payload normalization
- unified CLI help and command dispatch
- access control logic
- serial queue behavior
- queue batching behavior
- local bridge end-to-end execution
- static service configuration generation
- duplicate message handling

## Known Boundaries

- Service management currently targets macOS `launchd`.
- The bot currently focuses on text messages.
- This release is still local self-hosted, not cloud hosted.
- Rich cards, attachments, and multi-profile support are not part of this beta.

## 中文说明

`Feishu for Codex v0.3.0-beta` 聚焦把插件从本地 demo 推进为可控、可常驻运行的飞书协作入口。

### 新增

- 统一 CLI 入口：`npm run feishu -- <command>`。
- 长连接消息桥接：`message -> runner -> downstream command -> reply`。
- 仓库内置 runner，用于稳定承接 Feishu 到 Codex 的 payload 协议。
- 仓库内置 echo 脚本，用于接真实 Codex 前的本地桥接验证。
- 按 `chat_id` 或 `open_id` 隔离的会话状态持久化。
- 飞书聊天中的最小命令集：
  - `/help`
  - `/new`
  - `/status`
  - `/stop`
  - `/cd <path>`
- 访问控制：
  - owner
  - admins
  - allowed users
  - allowed chats
- 按会话串行执行的队列，以及可选的短窗口批处理。
- 基于飞书 `message_id` 的重复投递保护，避免重复本地执行和重复回复。
- macOS 优先的本地 service 管理：
  - `start`
  - `stop`
  - `restart`
  - `status`
- 基于 bot state 目录的 `launchd` plist 与 service 元数据持久化。

### Bot Bridge 能力

- 接收飞书长连接文本消息。
- 按群聊或发送者身份隔离会话状态。
- 通过稳定 runner 协议执行本地下游命令。
- 支持 `stdin` 和 `env` 两种 payload 模式。
- 持久化队列、工作区、运行状态和最近一次任务结果。
- 返回可读的拒绝、排队、取消、失败和完成提示。
- 在每个会话的最近消息窗口内忽略重复投递。

### Service 管理

- 在 bot state 目录下生成 `service/launchd.plist`。
- 写入 `service/service.json`，记录 label、mode、日志路径和时间戳。
- 默认使用：
  - `service/stdout.log`
  - `service/stderr.log`
- 以 `launchctl` 作为已加载和运行状态的事实来源。

- 当前 bot 主要聚焦文本消息
- 当前版本仍然是本地 self-hosted，不是云托管
- 富卡片、附件和多 profile 还不在这个 beta 中
