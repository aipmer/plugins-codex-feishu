# Feishu for Codex v0.3.2

Iteration release focused on productizing what was proven in real maintainer use today: bot diagnostics, private push defaults, and a lightweight daily digest path that does not expand the long-term maintenance surface too much.

## Added

- Lightweight digest entry:
  - `npm run feishu -- digest --preview`
  - `npm run feishu -- digest --send --confirm`
- Repo-provided digest wrapper script that reuses the existing Feishu project update sender.
- `/ids` command for copying current `chat_id` and `open_id` into allowlist config.
- Richer `/status` output with chat identity, runner command, bridge command, and local service summary.

## Changed

- Daily and weekly push defaults now favor Chinese output for real Feishu private assistant usage.
- The default digest template is now fully localized to Chinese.
- Documentation now recommends a docs-only macOS `launchd` example for personal scheduled digest sending, instead of building a second in-repo schedule subsystem.

## Fixed

- Corrected real maintainer testing after discovering the wrong Feishu `App ID` was being used during one validation pass.
- Added duplicate `message_id` handling to avoid repeated execution and repeated replies on Feishu retry delivery.
- Improved local operator clarity around bot runtime state versus `launchd` service state.

## Verification

Run:

```bash
scripts/smoke-test.sh
npm run feishu -- help
npm run feishu -- digest --preview
scripts/check-sensitive-values.sh
```

Real maintainer validation also included:

- a live Feishu bot bridge reply path
- a real `launchd` start/status/restart/stop cycle
- real private assistant push tests
- a real lightweight digest send

## Known Boundaries

- The digest flow still uses a fixed template as its content source.
- Personal scheduling is documented through macOS `launchd`, but not managed as a built-in subsystem.
- Rich cards, attachment handling, and multi-project digest aggregation remain outside this iteration.

## 中文说明

`Feishu for Codex v0.3.2` 聚焦把今天已经真实跑通的能力收口成更适合开源维护的产品迭代：更强的 bot 诊断能力、更清晰的私人推送默认路径，以及一个不会明显放大维护成本的轻量日报入口。

### 新增

- 轻量日报入口：
  - `npm run feishu -- digest --preview`
  - `npm run feishu -- digest --send --confirm`
- 仓库内置 digest 包装脚本，复用现有的飞书项目更新发送链路。
- `/ids` 命令，方便把当前 `chat_id`、`open_id` 直接复制进 allowlist 配置。
- 更丰富的 `/status` 输出，包含会话身份、runner command、bridge command 和本地 service 摘要。

### 调整

- 日报、周报推送默认更偏向中文输出，更适合真实飞书私人助理场景。
- 默认 digest 模板已经完整中文化。
- 对个人定时推送，文档改为推荐仓库外的 macOS `launchd` 示例，而不是继续把仓库扩展为第二套 schedule 子系统。

### 修复

- 在一次真实联调中定位到错误 `App ID` 配置后，重新校正了飞书机器人验证路径。
- 增加基于 `message_id` 的重复投递处理，避免飞书重试导致重复执行和重复回复。
- 提升本地运维视角下对 bot 运行状态与 `launchd` service 状态的区分度。

### 验证方式

```bash
scripts/smoke-test.sh
npm run feishu -- help
npm run feishu -- digest --preview
scripts/check-sensitive-values.sh
```

维护者真实联调还包括：

- 飞书 bot bridge 的真实回复链路
- 一次完整的 `launchd` start/status/restart/stop
- 私人助理真实推送测试
- 轻量 digest 的真实发送测试

### 当前边界

- digest 内容源当前仍然是固定模板。
- 个人定时推送目前只提供 macOS `launchd` 文档级接入，不做仓库内调度管理。
- 富卡片、附件处理和多项目摘要聚合还不在这一轮范围内。
